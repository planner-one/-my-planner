# 지원 공고 링크 Reader 동작 기록

_작성일: 2026-07-01_
_갱신일: 2026-07-04 (설계 감사 기준)_

이 문서는 지원 공고 페이지의 "링크로 초안 만들기" 기능이 어떻게 동작하는지 남겨두기 위한 작업 메모입니다.

## 목적

지원 공고 링크를 넣었을 때 회사명, 포지션, 플랫폼, 기술스택, 주요 업무, 소재지, 고용형태, 연봉/급여 같은 내용을 직접 옮겨 적지 않아도 초안으로 채우는 것이 목적입니다.

현재 구현은 무료/로컬 1차 기반입니다. 링크 분석 자체는 지원하지만, 배포 환경에서 안정적으로 외부 페이지를 읽으려면 추후 Firebase Functions 또는 별도 백엔드 API로 Reader API를 옮겨야 합니다.

2026-07-04 설계 감사 기준으로 이 Reader는 `PLANNER_SYSTEM_AUDIT_2026-07-04.md`의 보완점과 `UPDATE_SCHEDULE.md`의 v0.4.0 Integration Ready 후보에 연결되어 있습니다.

## 사용자 흐름

1. 사용자가 `지원 공고` 페이지를 엽니다.
2. `링크로 초안 만들기` 영역을 펼칩니다.
3. 기업 채용 페이지, Google Sites, 인크루트, 사람인, 원티드 같은 공고 링크를 입력합니다.
4. `분석해서 폼에 반영` 버튼을 누릅니다.
5. 앱이 링크 본문을 읽을 수 있으면 공고 내용을 분석해 작성 폼에 반영합니다.
6. 링크 본문 또는 Reader Markdown에서 이미지 후보를 찾으면 `페이지 이미지 후보`와 `첫 이미지 OCR` 버튼을 표시합니다.
7. 사용자가 이미지 OCR을 실행하면 이미지 글자를 `공고 원문/OCR` 칸에 붙이고, 다시 초안 필드를 보정합니다.
8. 회사, 포지션, 플랫폼, 상태, 마감일, 키워드, 추출 상세/메모를 사용자가 확인하고 보정합니다.
9. `추가`를 누르면 `jobPostings` 데이터로 저장됩니다.

링크만으로 본문을 못 읽는 경우에는 같은 영역의 페이지 내용 붙여넣기 칸이나 공고 원문/OCR 칸에 내용을 넣고 다시 반영합니다.

OpenAI API 결제 없이 현재 Codex를 활용하는 경우에는 `Codex로 정밀 정리` 보조 영역을 사용합니다.

1. 링크, 공고 원문/OCR, 기존 메모를 입력합니다.
2. `분석 템플릿 복사`를 눌러 Codex에 전달할 프롬프트를 복사합니다.
3. Codex가 회사, 포지션, 마감일, 주요 업무, 자격요건, 우대사항, 복지/혜택, 전형절차, 제출서류를 라벨별로 정리합니다.
4. 정리 결과를 앱의 붙여넣기 칸에 넣고 `결과를 폼에 반영`을 누릅니다.
5. 앱은 회사/포지션/마감/근무지/고용형태/키워드와 `추출 상세 / 메모`를 채웁니다.

이 흐름은 앱 안에서 OpenAI API를 호출하지 않고, API 키나 결제 수단도 필요로 하지 않습니다. 자동 분석 결과는 초안이므로 저장 전 사용자가 확인합니다.

## 코드 흐름

주요 파일:

- `src/pages/JobPostings.tsx`
- `src/services/jobPostingPageReader.ts`
- `src/utils/jobPostingDraft.ts`
- `vite.config.js`
- `scripts/check-job-posting-draft.mjs`

동작 순서:

1. `JobPostings.tsx`의 `applyLinkDraftToForm`이 입력 URL을 `normalizeJobUrl`로 정리합니다.
2. `getJobPostingPageText`가 먼저 같은 출처 API인 `/api/job-posting-page`를 호출합니다.
3. Vite 개발 서버의 `jobPostingPageApi` 미들웨어가 서버 쪽에서 대상 링크를 읽습니다.
4. Google Sites처럼 브라우저 fetch가 CORS로 막히기 쉬운 호스트는 Reader 경로를 우선 사용합니다.
5. 읽은 본문이 공고로 볼 만한 텍스트인지 `hasUsefulJobText`로 확인합니다.
6. `buildJobPostingLinkDraft`가 URL 정보와 읽은 본문, 사용자가 붙여넣은 보조 텍스트를 합쳐 초안을 만듭니다.
7. `inferJobPostingFromText`가 본문에서 회사명, 포지션, 마감일, 지역, 고용형태, 키워드, 메모를 추출합니다.
8. 같은 API 응답에 이미지 후보가 있으면 `imageUrls`로 화면에 전달합니다.
9. `첫 이미지 OCR`을 누르면 `/api/job-posting-image`가 이미지 파일을 같은 출처로 프록시하고, 브라우저의 `tesseract.js`가 OCR을 실행합니다.
10. 추출 결과는 작성 폼에 반영되고, 주요 업무와 조건은 `추출 상세 / 메모` 칸에 표시됩니다.

## 채워지는 필드

자동 반영 대상:

- `sourceUrl`: 원본 공고 링크
- `platform`: 사람인, 잡플래닛, 원티드, 점핏, 그룹바이, 인크루트, 기업/기관, 기타
- `company`: 기업명 또는 기관명
- `position`: 채용 직무/분야
- `deadline`: 지원 마감일
- `location`: 소재지 또는 근무지
- `employmentType`: 고용형태 또는 근무형태
- `keywords`: React, Next.js, Java, Spring Boot, AWS 같은 기술 키워드
- `note`: 사업내용, 채용 직무, 주요 업무, 기술스택, 소재지, 고용형태, 채용 인원, 연봉/급여, 지원 자격, 우대사항
- `nextAction`: 비어 있으면 `공고 확인 후 지원서 맞춤 수정` 기본값

마감일은 `마감`, `접수 기간`, `지원 기간`, `모집 기간`, `서류 접수`, `채용 기간`, `공고 기간`, `신청 기간` 같은 라벨 근처의 날짜만 사용합니다. 그래서 `업력: 2020.10.07` 같은 회사 정보 날짜는 마감일로 반영하지 않습니다.

## 사람인/원티드 처리 보강

v0.3.9 기준으로 사람인과 원티드는 플랫폼별 구조를 별도로 더 봅니다.

- 사람인: `Title: [회사] 공고명 - 사람인` 제목에서 회사명과 포지션을 우선 추출합니다. `경력/학력/근무형태/급여/근무지역`이 한 줄에 붙어 나오는 핵심 정보는 라벨 뒤 값을 잘라서 근무형태와 지역에 넣습니다. `시작일 ... 마감일 ...`처럼 한 줄에 날짜가 여러 개 있으면 `마감일` 뒤 날짜를 우선 사용합니다.
- 원티드: `[회사] 포지션 | 원티드` 제목과 `회사 ∙ 지역 ∙ 경력` 상단 정보를 봅니다. `주요업무`, `자격요건`, `근무지역`, `마감일` 섹션에서 직무 설명, 지역, 기술 키워드를 추출합니다.
- 키워드: React/Next.js/Java/Spring 같은 기본 기술 외에 AI, 백엔드, 프론트엔드, Wallet, Security, 블록체인, Python, Go, Kotlin, Rust, HSM, MPC, Secure SDLC, Threat Modeling 등을 인식합니다.
- 잡음 제외: 사람인 광고 상품, AI 서류합격 안내, 개인정보 문구, 원티드 저작권/무단전재 문구는 메모와 키워드 후보에서 제외합니다.
- 저장된 공고의 `분석 반영` 버튼도 이제 URL만 다시 해석하는 것이 아니라 같은 출처 Reader API로 링크 본문을 다시 읽어 placeholder 값을 보정합니다.

## Google Sites 처리

Google Sites는 브라우저에서 직접 `fetch`하면 CORS 정책 때문에 막히는 경우가 많습니다.

그래서 로컬 개발 환경에서는 다음 흐름을 사용합니다.

1. 브라우저가 외부 Google Sites를 직접 읽지 않습니다.
2. 브라우저는 같은 출처인 `/api/job-posting-page`만 호출합니다.
3. Vite 개발 서버가 서버 쪽에서 대상 페이지를 읽습니다.
4. 필요하면 Reader 텍스트를 사용합니다.
5. 브라우저는 정리된 JSON 결과만 받습니다.

이 방식은 로컬 Vite 개발 서버 미들웨어에 붙어 있습니다. Firebase Hosting 배포본에서는 Vite 미들웨어가 동작하지 않으므로 같은 기능을 운영 배포에서 유지하려면 Firebase Functions나 별도 백엔드 API가 필요합니다.

## 인크루트 기업 도메인 처리

`kdb.incruit.com`처럼 기업별 인크루트 도메인은 `https://...viewhire.asp?projectid=125`로 직접 접근하면 서버가 쿼리 없는 경로로 리다이렉트하며 "잘못된 경로" 응답을 줄 수 있습니다.

2026-07-04 기준 로컬 Reader는 인크루트 호스트에서 다음 순서로 보정합니다.

1. 브라우저 직접 fetch 대신 같은 출처 `/api/job-posting-page`를 우선 사용합니다.
2. Vite 미들웨어가 인크루트 링크를 Reader로 읽을 때 `https` 대상뿐 아니라 `http://kdb.incruit.com/...` 대상도 함께 시도합니다.
3. KDB 샘플 링크는 `http` 대상 Reader에서 `한국산업은행 채용정보`, `채용구분`, `담당직무`, `지원서 접수`, `보수수준`, `근 무 지` 본문을 읽습니다.
4. 파서는 이 라벨을 회사명, 직무, 마감일, 고용형태, 근무지, 메모로 반영합니다.

추가 보정:

- `/api/job-posting-page`가 없는 환경에서는 브라우저에서 Reader URL을 직접 한 번 더 시도합니다.
- 브라우저 CORS나 외부 Reader 실패로 본문을 읽지 못하면 URL 슬러그만으로 성공 처리하지 않습니다.
- `KDB`, `nextsystem`처럼 URL에서 추정한 값만 있는 경우에는 "초안 반영 성공"이 아니라 본문 추출 실패 안내와 붙여넣기 보정 흐름을 보여줍니다.

## 페이지 이미지 OCR 처리

일부 공고는 본문 텍스트가 아니라 이미지 안에 내용이 들어 있습니다.

현재 구현은 다음 범위까지 지원합니다.

1. `/api/job-posting-page`가 HTML의 `<img src="...">` 또는 Reader Markdown의 `![...](...)` 이미지 URL을 찾습니다.
2. 로고, 아이콘, 픽셀 이미지처럼 보이는 URL은 제외하고 최대 8개 후보를 반환합니다.
3. 화면에는 이미지 후보 개수와 `첫 이미지 OCR` 버튼이 표시됩니다.
4. 버튼을 누르면 `/api/job-posting-image`가 첫 번째 이미지 후보를 불러와 브라우저 OCR에 넘깁니다.
5. OCR 텍스트는 `공고 원문/OCR` 칸에 추가되고, 파서가 다시 회사명/직무/기술스택/메모를 보정합니다.

한계:

- 페이지 접근 자체가 막히면 이미지 후보도 찾을 수 없습니다.
- 이미지 URL이 JavaScript 실행 뒤에만 만들어지는 페이지는 현재 로컬 Reader API가 찾지 못할 수 있습니다.
- 인크루트 일부 기업 도메인처럼 서버가 직접 접근을 `잘못된 경로`로 처리하거나 Reader 결과가 비어 있는 링크는 이미지 OCR 후보가 나오지 않습니다.
- 여러 이미지 중 첫 번째 이미지가 실제 공고가 아닐 수 있으므로, 추후에는 이미지 후보 미리보기와 선택 기능이 필요합니다.

## 화면 표시 방식

지원 공고 작성 폼에는 두 개의 긴 입력 칸이 있습니다.

- `공고 원문/OCR`: 사용자가 직접 붙여넣은 원문이나 이미지 OCR 텍스트를 보관합니다.
- `추출 상세 / 메모`: Reader 또는 붙여넣기 분석으로 정리된 주요 업무, 기술스택, 조건을 보여줍니다.

저장된 공고 카드에서도 `공고 원문/OCR`과 `추출 상세 / 메모`를 다시 수정할 수 있습니다.

## 테스트 기준

`npm run check:job-draft`는 `scripts/check-job-posting-draft.mjs`를 실행합니다.

현재 확인하는 내용:

- Google Sites 링크만 있을 때 `nextsystem` 같은 경로명을 회사명으로 확정하지 않는지
- Google Sites 본문이 있을 때 `주식회사 스탁키퍼`를 회사명으로 잡는지
- `웹 (백엔드) / 웹 (프론트엔드)`를 포지션으로 잡는지
- 소재지, 고용형태, 기술스택을 추출하는지
- `업력: 2020.10.07`을 마감일로 오해하지 않는지
- 주요 업무와 기술스택이 `note`에 들어가는지
- 인크루트 기업 도메인에서 `KDB` 같은 서브도메인 기반 회사명을 잡는지
- KDB 인크루트 Reader 본문에서 `한국산업은행`, `은행업무 지원`, `2026-07-09`, `임시고용인력`, `본점, 국내지점, KDB Digital Square`를 추출하는지
- 사람인 Reader 제목/핵심 정보에서 회사명, 포지션, 근무형태, 근무지역, 마감일, AI/백엔드/프론트엔드 키워드를 추출하는지
- 원티드 공고에서 회사명, 보안/블록체인 포지션, 주요 업무, 지역, Wallet/Security/언어 키워드를 추출하고 `역` 같은 잘못된 한 글자 지역을 버리는지
- 광고, 개인정보, 저작권 문구가 `note`에 들어가지 않는지

## 한계

- 로그인, 쿠키, 권한이 필요한 페이지는 자동으로 읽기 어렵습니다.
- 이미지에만 정보가 있는 공고는 페이지 이미지 후보가 잡히면 `첫 이미지 OCR`을 시도할 수 있고, 후보가 없으면 이미지 파일 업로드 OCR 또는 직접 붙여넣기가 필요합니다.
- Reader 결과에는 사이트 네비게이션 텍스트가 섞일 수 있어 파서가 완벽하지 않습니다.
- AI가 아니라 규칙 기반 초안 생성이므로 저장 전 사용자가 반드시 확인해야 합니다.
- 로컬 Vite API는 배포 환경의 백엔드가 아닙니다.

## 추후 개선 후보

- Firebase Functions로 `/api/job-posting-page` 운영 API 이관
- 링크 페이지 내부 이미지 후보 미리보기와 사용자가 OCR할 이미지 선택
- AI 기반 구조화 추출로 마감일, 자격요건, 우대사항, 제출서류 정밀 분리
- 무료/유료 권한과 사용량 제한
- 원본 링크, 추출 텍스트, 사용자 보정 이력을 AI 맞춤 추천 데이터로 활용
