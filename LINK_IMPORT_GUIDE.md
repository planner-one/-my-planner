# 링크 자동 정리 가이드

_마지막 업데이트: 2026-07-04_

이 문서는 "링크 정리" 기능에 넣기 좋은 링크와, 자동 분석이 어려운 링크를 빠르게 구분하기 위한 메모입니다.

지원 공고 링크 Reader의 실제 동작 흐름과 검증 기준은 `JOB_POSTING_LINK_READER.md`에 별도로 정리합니다.
전체 설계 감사와 다음 개선 순서는 `PLANNER_SYSTEM_AUDIT_2026-07-04.md`, `UPDATE_SCHEDULE.md`를 함께 봅니다.

## 먼저 구분할 것

- Codex 세션: 공개 웹페이지라면 이 대화에서 직접 열어보고 내용을 요약하거나 추출할 수 있습니다.
- 현재 플래너 앱 1차 구현: 실제 웹페이지 본문은 직접 읽지 않지만, URL·사용자 메모·포스터 텍스트·업로드한 이미지 파일 OCR 결과를 기반으로 초안을 만듭니다.
- 지원 공고 페이지 1차 구현: Google Sites, 인크루트, 기업 채용 링크, 사람인, 원티드를 공고 링크로 인식하고, 붙여넣은 본문/OCR 텍스트나 Reader 본문에서 기업명·직무·소재지·고용형태·기술스택·주요 업무·연봉/급여를 추출합니다. 사람인/원티드는 제목, 핵심 정보, 근무지역, 마감일, 주요업무/자격요건 위치를 우선 보고 광고·개인정보·저작권 문구는 제외합니다. 로컬 개발 환경에서는 같은 출처 `/api/job-posting-page` Reader API로 Google Sites처럼 CORS가 막힌 공개 페이지를 읽고, 페이지 이미지 후보가 잡히면 `/api/job-posting-image`와 브라우저 OCR로 첫 이미지를 읽을 수 있습니다. 배포본에서 같은 기능을 유지하려면 이 API를 Firebase Functions 또는 별도 백엔드로 옮겨야 합니다.
- 추후 완성형 기능: 백엔드/API 또는 서버리스 함수가 공개 페이지 본문과 링크 페이지 안의 이미지를 읽고 AI가 제목, 일정, 마감일, 카테고리, 상태를 추출하는 방식이 필요합니다.

## 읽기 좋은 링크

다음 링크는 Codex가 읽기 쉽고, 추후 자동 정리 기능과도 잘 맞습니다.

- 로그인 없이 열리는 공개 공고 페이지
- 채용 공고, 교육 프로그램, 행사/세미나, 지원사업, 공모전 상세 페이지
- 제목, 모집 기간, 신청 마감일, 행사 일시, 장소, 주최 기관이 텍스트로 적힌 페이지
- 공공기관/학교/기업/청년 플랫폼의 일반 상세 페이지
- 공개 블로그, 뉴스, 공지사항, 자료 페이지
- 텍스트 선택이 가능한 공개 PDF

예시:

- `youth.seoul.go.kr`의 청년 지원사업/프로그램 상세 페이지
- `allforyoung.com`의 공모전/교육/행사 게시글
- 기업 채용 공고 상세 페이지
- 교육기관 프로그램 모집 페이지
- 행사 등록 안내 페이지

## 이미지 포스터 링크

이미지 포스터형 공고도 지원 대상입니다. 다만 일반 텍스트 페이지와 달리 OCR과 이미지 분석이 필요합니다.

- 카드뉴스, 포스터, 홍보 이미지에 제목/일정/마감/장소가 들어간 게시글
- Instagram, 블로그, 공공기관 공지처럼 이미지가 핵심 정보인 게시글
- 텍스트 설명은 짧고 포스터 이미지에 세부 내용이 들어간 모집 공고

처리 방향:

- 업로드한 이미지 파일에서 글자를 OCR로 추출합니다.
- 추출한 텍스트에서 제목, 일정, 마감일, 장소, 카테고리를 다시 분석합니다.
- 이미지 글자는 오인식될 수 있으므로 저장 전 미리보기와 사용자 확인이 반드시 필요합니다.
- 현재 플래너 1차 구현은 이미지 파일 OCR 또는 직접 붙여넣은 포스터 텍스트에서 신청 마감일, 행사일, 시간, 장소를 초안으로 추출합니다.
- 링크 페이지 안의 이미지 URL이 HTML이나 Reader Markdown에 노출되면 로컬 개발 환경에서는 첫 이미지 OCR을 시도할 수 있습니다. JavaScript 실행 뒤에만 이미지가 생기거나 접근 제한이 있는 페이지는 백엔드/API 또는 이미지 분석 모델 연결이 필요합니다.
- 브라우저 OCR은 첫 실행 시 언어 데이터를 불러오느라 시간이 걸릴 수 있고, 네트워크 상태에 따라 실패할 수 있습니다.

## 읽기 어려운 링크

다음 링크는 본문 자동 분석이 어렵거나 수동 보정이 필요합니다.

- Google Form처럼 로그인, 쿠키 허용, 접근 권한이 필요한 링크
- 네이버 카페, 일부 커뮤니티, 사내 게시판처럼 회원 로그인 뒤에 보이는 글
- Instagram, LinkedIn 등 본문을 제한적으로 노출하거나 로그인 유도가 강한 서비스
- 이미지 포스터만 있고 페이지 본문 텍스트가 없는 경우. 현재 1차 기능에서는 이미지 파일을 업로드하거나 포스터 내용을 직접 붙여넣어 초안을 만듭니다.
- JavaScript 렌더링이 강하고 서버에서 본문 HTML을 거의 주지 않는 페이지
- 서버 직접 접근 시 `잘못된 경로`처럼 차단 응답을 주거나 Reader 결과가 비어 있는 채용 페이지
- 결제/회원 전용 자료, 비공개 문서, 권한이 필요한 Google Docs/Sheets/Drive 링크
- 단축 URL만 있고 최종 페이지가 불명확하거나 접근 제한이 있는 링크

## 잘 쓰는 방법

- 링크만 넣어도 되지만, 중요한 내용이 보이면 메모에 "마감 7/20, 온라인, 교육"처럼 같이 적으면 초안 정확도가 좋아집니다.
- 이미지 포스터는 자료 유형을 "이미지 포스터"로 바꾸고 이미지 파일을 선택하거나, 포스터에서 읽은 제목/모집 기간/행사일/장소를 붙여넣으면 더 유용합니다.
- 자동 추출 결과는 반드시 미리보기에서 확인한 뒤 저장합니다.
- OCR이 실패하거나 글자가 깨져 보이면 포스터 내용을 직접 수정한 뒤 다시 초안을 만듭니다.
- Google Form은 신청 링크로 보존하고, 실제 제목/마감일/일정은 원 공고 페이지에서 가져오는 방식이 좋습니다.
- 가능하면 Google Form 단독 링크보다 원본 모집 공고 링크를 먼저 넣습니다.

## 추후 구현 메모

- 공개 페이지 본문 읽기, PDF 텍스트 추출, AI 분석, 사용량 제한, 유료 권한 처리는 백엔드/API 또는 Firebase Functions 같은 서버 측 처리가 필요합니다.
- 링크 페이지 내부 이미지 자동 수집, 고정밀 이미지 분석, 대량 OCR은 모델 비용이 발생할 수 있어 유료 기능 범위와 사용량 제한을 함께 정해야 합니다.
- 외부 페이지의 robots, 약관, 개인정보, 저작권 범위를 확인하고 과도한 수집은 피해야 합니다.
- 분석 실패 시에도 원본 URL과 사용자가 입력한 메모는 보존해야 합니다.
- 운영 배포에서 링크 본문 Reader를 안정적으로 쓰려면 `UPDATE_SCHEDULE.md`의 v0.4.0 Integration Ready 범위에서 Firebase Functions 또는 별도 백엔드/API 이관을 먼저 결정해야 합니다.
