# 모바일 탐색·테마·반응형 밀도 개선 설계

_작성일: 2026-07-16_

## 목표

모바일 하단 메뉴가 브라우저 주소창 상태와 무관하게 항상 보이도록 고치고, 사용자가 로그인 계정별로 원하는 바로가기 네 개를 선택하고 순서를 정할 수 있게 한다. 모바일 상단에는 네 가지 테마를 고르는 진입점을 추가하며, 화면 밀도는 휴대폰·태블릿·웹의 실제 가용 폭에 맞춰 자동 적용하되 기존 웹 수동 배율을 보존한다.

## 합의된 범위

1. 하단 메뉴는 `홈`을 첫 번째에 고정하고 사용자가 고른 네 개를 뒤에 배치해 총 다섯 칸으로 표시한다.
2. 기본 선택은 `캘린더`, `작업 관리`, `기회 일정`, `프로필`이다.
3. 모바일 전체 메뉴에서 `하단 메뉴 편집`을 열어 네 개를 선택하고 위·아래 버튼으로 순서를 바꾼다.
4. 모바일 상단 오른쪽의 팔레트 버튼에서 `라이트`, `다크`, `코랄`, `블루`를 직접 선택한다.
5. 화면 밀도는 기본적으로 자동이며 휴대폰 767px 이하와 태블릿 768~1199px은 100%, 웹 1200px 이상은 90%를 사용한다.
6. 기존 80~110% 수동 배율은 웹의 `직접 설정` 값으로 유지한다. 휴대폰·태블릿은 글자와 44px 터치 영역을 보호하기 위해 100%를 유지한다.
7. 기존 일일 루틴 안정화, 모바일 로그아웃, 모바일 입력 보완과 48열 대시보드 좌표를 그대로 유지한다.
8. 타입 검사, 회귀 검사, 프로덕션 빌드와 Firebase Hosting 배포까지 이번 작업 범위에 포함한다.

## 확인된 원인과 현재 한계

### 하단 메뉴가 늦게 보이는 원인

모바일 CSS가 `--app-viewport-height`를 `100vh`로 강제한다. Android Chrome에서 주소창이 보이는 동안 `100vh`는 실제 가시 높이보다 클 수 있다. `PageShell`은 이 값만큼 높이를 차지하고 본문 뒤에 하단 메뉴를 배치하므로, 주소창이 접혀 가시 영역이 커질 만큼 스크롤해야 하단 메뉴가 나타난다.

### 하단 메뉴가 고정된 원인

`PageShell`의 `BOTTOM_TABS` 상수에 여섯 페이지가 고정되어 있고 렌더링도 이 상수만 필터링한다. 사용자 설정과 저장 경계가 없다.

### 화면 밀도 불일치

현재 `uiScale`은 계정에 숫자 하나로 저장되고 `body zoom`에 적용된다. 767px 이하에서는 CSS가 100%로 덮어쓰기 때문에 모바일 프로필의 슬라이더 값과 실제 화면이 다를 수 있다. 반대로 768px 이상 태블릿은 웹용 축소값을 즉시 적용받아 글자와 터치 영역이 작아질 수 있다.

## 검토한 접근

### 모바일 하단 메뉴

1. 기존 여섯 칸을 모두 교체 가능하게 유지: 변경량은 작지만 360~390px에서 긴 라벨과 터치 영역이 계속 좁다.
2. `홈` 고정과 사용자 선택 네 칸으로 총 다섯 칸 구성: 위치 예측 가능성과 선택권을 함께 확보한다. 이 방식을 채택한다.
3. 가로 스크롤 바로가기: 많은 메뉴를 담을 수 있지만 위치가 예측되지 않고 접근성이 떨어져 채택하지 않는다.

### 설정 저장

1. `localStorage`: 빠르지만 기기 간 동기화가 되지 않고 계정 전환 시 혼동될 수 있다.
2. Firestore의 `users/{uid}` 선택 설정: 기존 자동 저장과 계정 범위를 재사용하고 다른 기기에서도 유지된다. 이 방식을 채택한다.
3. Firestore와 별도 로컬 캐시 이중화: 현재 범위에는 동기화 복잡도가 커서 채택하지 않는다.

### 화면 밀도

1. 기존 단일 슬라이더 유지: 모바일 표시값과 실제 적용값이 달라지는 문제를 남긴다.
2. 완전 자동만 제공: 단순하지만 웹에서 사용자의 시력과 선호를 반영할 수 없다.
3. 자동을 기본으로 하고 웹 직접 설정을 보존: 휴대폰·태블릿 접근성과 웹 개인화를 함께 유지한다. 이 방식을 채택한다.

## 화면과 상호작용 설계

### 항상 보이는 하단 메뉴

- 모바일 앱 높이는 동적 가시 높이인 `100dvh`를 사용한다. 지원 환경을 위한 `100vh` 선언 뒤에 `100dvh`를 둔다.
- 하단 메뉴는 앱 셸의 축소되지 않는 flex 자식으로 두고 `flex-shrink: 0`과 하단 safe area를 적용한다.
- 본문만 내부 스크롤하며 하단 메뉴를 `position: fixed`로 겹치지 않는다.
- `viewport-fit=cover`를 viewport meta에 추가해 iOS safe area 값을 사용할 수 있게 한다.

### 하단 메뉴 편집

- 하단 바의 첫 칸은 항상 `홈`이며 제거하거나 이동할 수 없다.
- 나머지 네 칸은 계정 설정 순서대로 렌더링한다.
- 모바일 전체 메뉴 상단의 눈에 보이는 `하단 메뉴 편집` 버튼으로 전용 편집 패널을 연다. 긴 전체 메뉴의 마지막까지 스크롤해야 편집할 필요가 없게 한다.
- 편집 패널은 선택된 네 항목을 먼저 보여주고 각 항목에 `위로`, `아래로`, `제거` 버튼을 제공한다. 선택 가능한 전체 페이지는 그룹별로 보여주고 `추가` 버튼을 제공한다.
- 선택 개수는 정확히 네 개로 유지한다. 네 개가 선택된 상태에서 다른 항목을 추가하면 먼저 하나를 제거하라는 안내를 표시한다.
- `완료`를 누르면 즉시 하단 메뉴에 반영하고 계정 저장 흐름을 실행한다. `초기화`는 기본 네 항목으로 되돌린다.
- 모든 버튼은 최소 44px 터치 영역, 보이는 한국어 라벨, 키보드 포커스와 상태 안내를 제공한다. 드래그만으로 순서를 바꾸는 방식은 사용하지 않는다.

### 상단 테마 선택

- 모바일 상단 오른쪽에 44px 팔레트 아이콘 버튼을 추가한다. 데스크톱은 기존 사이드바 테마 UI를 유지한다.
- 버튼을 누르면 현재 테마가 표시된 네 가지 선택 패널을 연다.
- 테마를 선택하면 즉시 적용하고 기존 `theme` localStorage 저장 방식을 재사용한 뒤 패널을 닫는다.
- 패널은 바깥 클릭과 Escape로 닫히고, 열릴 때 내부로 포커스를 이동하며 닫힐 때 팔레트 버튼으로 포커스를 돌려준다.

### 반응형 화면 밀도

- 기기 이름이나 User-Agent 대신 현재 CSS viewport 폭을 사용한다. 회전, 분할 화면, 창 크기 변경도 같은 기준으로 즉시 재계산한다.
- `자동(권장)` 모드:
  - 767px 이하: 휴대폰 100%
  - 768~1199px: 태블릿 100%
  - 1200px 이상: 웹 90%
- `직접 설정`은 웹 1200px 이상에서 기존 `uiScale` 80~110% 값을 적용한다. 휴대폰·태블릿에서는 100%를 유지하고 저장된 웹 수동값을 지우지 않는다.
- 프로필의 `화면 밀도` 영역은 `자동(권장)`과 `직접 설정`을 제공한다. 현재 화면 분류와 실제 적용값을 `현재: 태블릿 · 100%`처럼 표시한다.
- 직접 설정 슬라이더는 웹에서만 활성화한다. 휴대폰·태블릿에서는 `터치와 글자 가독성을 위해 100%가 적용됩니다`라고 설명한다.
- 기존 사용자에게 `densityMode`가 없으면 자동으로 해석하고 기존 `uiScale`은 나중에 직접 설정을 선택할 때 사용할 값으로 보존한다.

## 데이터 구조와 하위 호환성

`UserData`에 선택 설정을 추가한다.

```ts
interface NavigationPreferences {
  mobileBottomTabs: string[]
  updatedAt: string
}

type DensityMode = 'auto' | 'manual'

interface DisplayPreferences {
  densityMode: DensityMode
  updatedAt: string
}
```

- `navigationPreferences.mobileBottomTabs`에는 `dashboard`를 제외한 네 페이지 ID만 순서대로 저장한다.
- 로드할 때 허용된 페이지인지 검사하고, 중복과 `dashboard`를 제거하고, 최대 네 개로 자른 뒤 부족한 항목은 기본값으로 보충한다.
- 설정이 없는 기존 사용자는 기본 네 항목과 `auto` 밀도를 사용한다.
- `uiScale` 필드는 삭제하거나 의미를 바꾸지 않고 웹 수동값으로 보존한다.
- 오래 열린 탭의 저장이 새 설정을 덮지 않도록 `updatedAt`이 더 최신인 설정 객체를 선택하는 병합 규칙을 추가한다.
- 테마는 기존과 동일하게 현재 브라우저의 localStorage에 저장하며 Firestore 구조를 늘리지 않는다.

## 오류와 경계 처리

- 저장값에 삭제되거나 알 수 없는 페이지 ID가 있으면 조용히 제거하고 기본값으로 보충한다.
- 선택이 네 개보다 적으면 완료할 수 없게 하고 필요한 개수를 안내한다.
- 저장이 실패하면 편집 패널을 유지하고 실패 안내와 재시도 버튼을 제공한다.
- 로그아웃과 데이터 저장 중에는 기존 중복 실행 방지 흐름을 유지한다.
- 테마 선택 실패 가능성이 없는 localStorage 적용은 즉시 반영하되 저장 예외가 나도 현재 세션 테마는 유지한다.
- 밀도 모드 계산에 잘못된 수동값이 들어오면 80~110 범위로 보정한다.

## 검증 기준

- Android 주소창이 열린 모바일 모델에서 앱 높이가 `100dvh`이고 하단 메뉴가 가시 영역 안에 유지되는 CSS 계약 검사
- 360px과 390px에서 하단 메뉴 다섯 칸, 44px 터치 영역, safe area 확인
- 기본값, 잘못된 ID, 중복, 초과, 부족 저장값을 네 개로 정규화하는 실행형 단위 검사
- 홈 고정, 최대 네 개 선택, 순서 이동과 초기화 검사
- 설정의 최신 `updatedAt` 병합 검사
- 테마 패널의 네 옵션, 현재 선택, Escape·포커스 복귀 계약 검사
- 자동 밀도 계산: 390px 100%, 767px 100%, 768px 100%, 1199px 100%, 1200px 90%
- 웹 직접 설정의 80~110 보정과 휴대폰·태블릿 100% 보호 검사
- 기존 루틴 크기, 로그아웃, 사용자 데이터 병합, 온보딩, Todo, 브리핑, 기회 일정, 내 신청 회귀 검사
- TypeScript 검사, 문서 동기화, 프로덕션 빌드와 Firebase Hosting 운영 자산 확인

## 범위 제외

- 모바일 대시보드의 48열 드래그·리사이즈 편집 재설계
- 모든 페이지의 브레이크포인트를 하나로 통합하는 대규모 CSS 재작성
- 테마의 계정 간 동기화
- 하단 메뉴의 드래그 전용 순서 변경
- React Grid Layout `1.4.4` 변경
- 앱 버전 변경
